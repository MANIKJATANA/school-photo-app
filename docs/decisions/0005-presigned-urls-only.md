# ADR 0005: All photo bytes flow through S3 presigned URLs; the API never proxies image data

- **Status**: Accepted
- **Date**: 2026-05-08
- **Deciders**: Project owner

## Context

Photos can be large (multi-megabyte JPEGs). Schools may upload thousands per event. If the API proxies bytes:

- Java request threads tie up on uploads/downloads (memory + connection pressure).
- Egress cost is paid twice — once into the API, once back out.
- Multipart upload tuning becomes the API's problem.

S3 presigned URLs let the client `PUT`/`GET` directly against S3. The API only generates short-lived signed URLs and validates metadata.

## Decision

All photo data is moved between client and S3 via **presigned URLs only**. The API never reads or writes image bytes.

Upload flow:
1. `POST /photos:initiate-upload` → server creates `photo` row in `PENDING`, returns `{photo_id, presigned_put_url, key, expires_in: 600}`.
2. Client `PUT`s bytes to S3.
3. `POST /photos/{id}:confirm-upload` → server `HEAD`s the object (size sanity), flips `upload_status = UPLOADED`, writes outbox `PHOTO_UPLOADED`.
4. Sweeper deletes `PENDING` rows older than 1h.

Download flow: API returns presigned `GET` URLs (≤5-min TTL) per request. FE caches images locally with bounded LRU + infinite scroll.

S3 access is wrapped in a `BlobStore` interface (`putUrl`, `getUrl`, `head`, `delete`). The S3 impl is the only place `S3Client`/`S3Presigner` is referenced.

## Consequences

### Positive

- API is stateless w.r.t. image bytes. Horizontal scale untouched by photo volume.
- Egress paid once — directly between client and S3.
- Storage backend is swappable (S3 today, MinIO/GCS/R2 tomorrow) via `BlobStore`.

### Negative

- Two-step upload (initiate + confirm) is more API surface than a single multipart upload.
- A confirm step that's missed leaves orphan rows; the sweeper handles them.
- Presigned URLs are bearer tokens for bounded time — TTL discipline is now a security concern (ADR-0006 boundary, security-reviewer enforces).

### Neutral

- Thumbnails (Phase 6) follow the same pattern — generated by a worker that reads/writes through `BlobStore`.

## Alternatives considered

- **API-proxied uploads** — rejected: thread pressure, double egress.
- **Direct S3 uploads with public-read ACLs** — rejected: leaks photos.
- **CloudFront signed cookies for downloads** — interesting for a future CDN layer, but premature for v1.
